# OpenProject Deployment with Kustomize

This directory contains the configuration for deploying OpenProject using Helm + Kustomize to work around the init-container tmp volume mounting issue.

## Problem Solved

The OpenProject Helm chart (GitHub issue #264) doesn't apply `extraVolumeMounts` to init-containers. This means:

- ✅ Main containers can use hostPath volumes (via `extraVolumeMounts`)
- ❌ Init-containers (`wait-for-db`) cannot use hostPath volumes
- ❌ With `readOnlyRootFilesystem: true`, init-containers crash because they can't write to `/tmp`

This kustomization.yaml patches the Helm-rendered manifests to add hostPath volumes to init-containers.

## File Structure

```
/app-storage/k3s/openproject/
├── kustomization.yaml          # Kustomize configuration (patches Helm output)
├── values.yml                  # Helm values file
└── rendered-manifests.yaml     # Generated by helm template command
```

## Prerequisites

### 1. Add Helm Repository

```bash
helm repo add openproject https://charts.openproject.org
helm repo update
```

### 2. Prepare HostPath Directories on Node

SSH into `andromeda-alpheratz` and create directories with correct permissions:

```bash
# Create directories
sudo mkdir -p /tmp/openproject-tmp /tmp/openproject-app-tmp

# Set ownership to UID 1000 (OpenProject user)
sudo chown -R 1000:1000 /tmp/openproject-tmp /tmp/openproject-app-tmp

# Set permissions with sticky bit (important for /tmp directories)
sudo chmod 1777 /tmp/openproject-tmp /tmp/openproject-app-tmp
```

### 3. Install Kustomize

```bash
# On Linux/Mac
curl -s "https://raw.githubusercontent.com/kubernetes-sigs/kustomize/master/hack/install_kustomize.sh" | bash

# Or using package manager
sudo apt-get install kustomize  # Ubuntu/Debian
brew install kustomize              # macOS
```

## Deployment Steps

### Step 1: Render Helm Templates

```bash
cd /app-storage/k3s/openproject

# Render OpenProject Helm chart to YAML
helm template openproject openproject/openproject \
  -f values.yml \
  --namespace collab \
  > rendered-manifests.yaml
```

### Step 2: Apply with Kustomize

```bash
# Build and apply patched manifests
kustomize build . | kubectl apply -f -
```

### Step 3: Verify Deployment

```bash
# Check all pods are running
kubectl get pods -n collab -l app.kubernetes.io/name=openproject

# Check init-container logs
kubectl logs -n collab -l app.kubernetes.io/name=openproject -c wait-for-db

# Check main container logs
kubectl logs -n collab -l app.kubernetes.io/component=web -c openproject
```

## What the Kustomize Patches Do

The kustomization.yaml applies these patches:

### 1. Web Deployment (`openproject-web`)
- ✅ Replaces init-container `/tmp` and `/app/tmp` mounts with hostPath volumes
- ✅ Adds hostPath volumes to pod spec

### 2. Worker Deployment (`openproject-worker-default`)
- ✅ Replaces init-container `/tmp` and `/app/tmp` mounts with hostPath volumes

### 3. Cron Deployment (`openproject-cron`)

### 4. Seeder Job (`openproject-seeder`)

## Troubleshooting

### Problem: "mountPath must be unique" Error

**Error**: `spec.template.spec.initContainers[0].volumeMounts[2].mountPath: Invalid value: "/tmp": must be unique`

**Cause**: Using `op: add` instead of `op: replace` in kustomize patches

**Solution**: This kustomization.yaml uses `op: replace`, which is correct.

### Problem: Permission Denied on HostPath

**Error**: `Error: EACCES: permission denied, mkdir '/tmp/...'`

**Cause**: HostPath directories owned by root, not UID 1000

**Solution**: On the node, run:
```bash
sudo chown -R 1000:1000 /tmp/openproject-tmp /tmp/openproject-app-tmp
sudo chmod 1777 /tmp/openproject-tmp /tmp/openproject-app-tmp
```

### Problem: Pod Still Crashing

**Check logs**:
```bash
kubectl logs -n collab -l app.kubernetes.io/name=openproject --all-containers=true
```

**Common issues**:
- Database not ready (check postgres connection)
- S3 not configured (if enabled)
- Permission issues (verify hostPath ownership)

## Notes

### Why Use `op: replace` Instead of `op: add`?

The Helm chart already includes `/tmp` and `/app/tmp` volumeMounts in init-containers (even with `useTmpVolumes: false`). Using `op: add` would create duplicate mount paths, causing "mountPath must be unique" errors.

Using `op: replace` replaces the entire volumeMounts array, ensuring only the hostPath volumes are present.

### Security Considerations

- HostPath volumes give pods access to node's filesystem
- Ensure hostPath directories are properly secured (1777 permissions with sticky bit)
- This setup is tied to specific node (`andromeda-alpheratz`) via nodeSelector

### Alternative Approaches

1. **Fork the Helm Chart**: Modify chart templates to include `extraVolumeMounts` in init-containers
2. **Use Ephemeral Volumes**: Revert to using PVCs with local-path storage (requires fixing permissions on node)
3. **Development Mode**: Set `develop: true` in values.yml (not recommended for production)
